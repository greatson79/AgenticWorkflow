## Fork-Based Sessions Branch Strategy (확장 기능)

이 프레임워크는 `/fork` 기능을 활용하여 **기술과 개발 노하우의 여러 경로를 동시에 탐색**합니다.
이를 통해 기술 선택의 깊이 있고 견고한 기초를 확보합니다.

### 전략 개요

```
초기 설정 (리서치 대상 + 주제 + 영역)
  ↓
[PHASE 1: 기술 심층 조사 - 10개 Branch 병렬 실행]
  ├─ /fork Branch 1.1: Core Tech Researcher (공격적 기술 선택)
  ├─ /fork Branch 1.2: Core Tech Researcher (보수적 기술 선택)
  ├─ /fork Branch 2.1: Architecture Specialist (점진적 확장 설계)
  ├─ /fork Branch 2.2: Architecture Specialist (초기 완성도 설계)
  ├─ /fork Branch 3.1: Dev Workflow Expert (빠른 개발 프로세스)
  ├─ /fork Branch 3.2: Dev Workflow Expert (견고한 개발 프로세스)
  ├─ /fork Branch 4.1: Technical Debt Manager (기술 부채 최소화)
  ├─ /fork Branch 4.2: Technical Debt Manager (실용적 부채 관리)
  ├─ /fork Branch 5.1: Theory Foundation Expert (최신 이론 기반) ← NEW
  └─ /fork Branch 5.2: Theory Foundation Expert (고전 이론 기반) ← NEW
  ↓
[PHASE 2: 기술 토론]
  ├─ /fork Branch 2.A: 최신 기술 우선 토론
  ├─ /fork Branch 2.B: 안정성 우선 토론
  ├─ /fork Branch 2.C: 개발 속도 우선 토론
  └─ /fork Branch 2.D: 유지보수성 우선 토론
  ↓
[PHASE 3: 기술 구현 시나리오]
  ├─ /fork Branch 3.A: Cutting Edge (최신 기술, 높은 리스크)
  ├─ /fork Branch 3.B: Balanced-Tech (기술과 현실의 균형)
  └─ /fork Branch 3.C: Proven Stack (검증된 기술, 낮은 리스크)
  ↓
[PHASE 4: 기술 로드맵 최종 확정]
  → 선택된 시나리오의 기술 스택 상세화
  → 개발 프로세스 정의
  → 기술 부채 관리 계획
```

### Branch 사용의 목적

**조사 Phase에서의 Branch**
- 같은 기술 영역에서도 다양한 선택이 가능함을 인정
- 예: "최신 프레임워크 채택" vs "검증된 프레임워크 선택"
- 각 선택의 장단점을 명확히 검증

**토론 Phase에서의 Branch**
- 어떤 기술 철학을 우선시하냐에 따라 결론이 달라짐을 검증
- "최신 기술" vs "안정성" vs "개발 속도" vs "유지보수성"의 경로
- 각 경로의 비용-효과를 파악

**시나리오 Phase에서의 Branch**
- 최신 기술 vs 균형 vs 검증된 기술 3가지 구현 경로 동시 검증
- "만약 우리가 더 과감했다면?" "더 신중했다면?"을 동시에 평가
- 최종 기술 선택의 근거를 명확히 함

## 실행 제약 (Constraints)

이 리서치 프로세스를 진행할 때 다음 제약을 유지하세요:

**조사 단계 제약**
- 각 Teammate는 **근거 없는 기술 추천 금지** (유행만으로는 안 됨)
- 해당 기술을 사용하는 **실제 사례 최소 3개 분석** (회사 규모, 도메인 유사도)
- "이 기술은 안 된다"는 결론은 **구체적 이유 3가지 이상** 제시
- 각 Teammate는 다른 관점의 영향을 받지 말 것 (기술적 객관성 유지)
- **Theory Foundation Expert는 반드시 이론적 근거를 명시**할 것
  - 고전 이론: 원저자, 논문/책, 발표년도
  - 최신 이론: 최근 발전, 새로운 개념, 비판 논의
- **이론과 실무의 갭을 명확히 할 것** (이론은 이상, 현실은 현실)

**토론 단계 제약**
- 토론은 **기술 선택의 근거**에만 집중 (경영 논리 아님)
- "좋아 보인다"는 주장은 안 됨 → "왜 좋은가" 기술적 근거 필수
- 한 기술이 연속 3번 이상 강조되면, 대안 기술에 명시적 발언 기회 제공
- **모든 권장사항은 성능 수치, 벤치마크, 또는 실제 사례**로 뒷받침

**시나리오 단계 제약**
- Cutting Edge: **최신 기술만 선택** (1-2년 이내 출시, 커뮤니티 활발)
- Balanced-Tech: **검증 + 최신의 균형** (3-5년 사용 사례, 대규모 채택)
- Proven Stack: **5년 이상 검증된 기술만** (엔터프라이즈 검증됨)
- 각 시나리오의 **개발 난이도, 학습곡선, 채용 난이도** 명시

## 성공 지표 (Success Points)

이 기술 리서치 프로세스가 **성공했는지** 판단하는 기준:

### Phase 1 (기술 조사) 성공 지표

✅ **각 Teammate별 기술 분석이 깊이 있는가?**
- [ ] 단순 나열이 아니라, 각 기술의 **구체적 장단점**이 명시되는가?
- [ ] 근거가 명확한가? (성능 수치, 실제 사례, 아키텍처 다이어그램)
- [ ] 우리 환경에 맞을지에 대한 **현실성 평가**가 있는가?

✅ **10개 Branch가 기술적으로 의미 있는 대안을 제시하는가?**
- [ ] 공격적 vs 보수적 선택이 **기술 차원**에서 명확히 다른가?
- [ ] 각 Branch의 결론이 **상충**하는가? (충돌 없으면 분석 부족)
- [ ] 학습곡선, 성능, 확장성, 안정성 등 **다각도 평가**가 있는가?

✅ **이론적 기초가 탄탄한가? (Theory Foundation Expert 평가)**
- [ ] **고전 이론 분석**: 기술의 근본 설계 철학이 명확한가?
  - 원저자/연구자: [명시되어 있나?]
  - 핵심 원칙: [불변 원칙이 무엇인가?]
  - 검증 기간: [얼마나 오래 검증되었나?]

- [ ] **최신 이론 분석**: 최근 기술 발전의 이론적 배경이 명확한가?
  - 새로운 개념: [무엇이 새로운가?]
  - 이전 한계 극복: [무엇을 극복했나?]
  - 새로운 한계: [무엇이 문제인가?]

- [ ] **이론과 실무의 갭**: 이론을 실무에 적용할 때의 주의점이 있는가?
  - "이론대로 하면 X인데, 현실은 Y다"의 설명
  - 타협점: [어디까지 이론을 따를 것인가?]

✅ **구현 현실성이 평가되는가?**
- [ ] 각 기술의 우리 팀이 **실제로 배울 수 있는가?**
- [ ] 채용 시장에서 **이 기술의 엔지니어를 구할 수 있는가?**
- [ ] 6개월/12개월 개발 일정에 **현실적으로 맞는가?**

### Phase 2 (기술 토론) 성공 지표

✅ **기술 간 트레이드오프가 명확히 드러났는가?**
- [ ] 최소 3개 이상의 **기술적 충돌** 포착됨
  - 예: 성능 vs 개발 속도, 확장성 vs 단순성
- [ ] 각 충돌마다 **정량적 비교**가 이루어졌는가?
  - 예: "A는 쿼리 응답 100ms, B는 50ms" (성능 벤치마크)
- [ ] "기술 철학의 차이"를 명확히 했는가?
  - 예: "마이크로서비스는 초기 복잡도 높지만 확장에 유리"

✅ **실무 경험이 반영되는가?**
- [ ] 각 기술을 **실제로 사용해본 엔지니어의 의견** 포함
- [ ] "이 기술로 프로덕션 서비스 돌린 사람"의 조언이 있는가?
- [ ] 실패 사례와 해결 방법이 포함되는가?

✅ **합의 영역이 명확해졌는가?**
- [ ] 4개 관점(최신, 안정, 속도, 유지보수)이 **모두 동의하는 기술** 파악됨
- [ ] 그 기술들의 조합이 **실제 아키텍처**를 이룰 수 있는가?

### Phase 3 (시나리오) 성공 지표

✅ **3가지 기술 시나리오가 **현실적으로 구분**되는가?**
- [ ] Cutting Edge: "최신 기술을 실전에 적용하려면?"의 시뮬레이션
  - 개발 시간: [예상 +30%]
  - 버그 위험: [높음]
  - 학습 필요: [매우 높음]

- [ ] Balanced-Tech: "기술과 현실의 균형"의 시뮬레이션
  - 개발 시간: [기준]
  - 버그 위험: [중간]
  - 학습 필요: [중간]

- [ ] Proven Stack: "검증된 기술로 최대한 빠르게"의 시뮬레이션
  - 개발 시간: [예상 -20%]
  - 버그 위험: [낮음]
  - 학습 필요: [낮음]

✅ **각 시나리오의 **구체적 기술 스택**이 있는가?**
- [ ] 각 시나리오마다 **Frontend, Backend, Database, DevOps 등** 모두 정의됨
- [ ] 각 계층의 기술이 **서로 호환**되는가?
- [ ] 각 조합이 **실제로 프로덕션에서 작동**할 것 같은가?

✅ **각 시나리오의 **구체적 개발 계획**이 있는가?**
- [ ] 각 시나리오별 **6개월 개발 일정** 제시
- [ ] 각 시나리오별 **팀 구성** (프론트 vs 백엔드 vs DevOps)
- [ ] 각 시나리오별 **리스크와 대응책**

### Phase 4 (최종 기술 로드맵) 성공 지표

✅ **선택된 시나리오의 기술 스택이 **상세하고 실행 가능한가?**
- [ ] 전체 아키텍처 다이어그램이 있는가?
- [ ] 각 계층의 **구체적 기술과 버전**이 명시되는가?
  - 예: "Node.js v18 + Express v4 + PostgreSQL v14"
- [ ] **API 설계, DB 스키마, 인프라** 등이 대략적으로 정의되는가?

✅ **개발 프로세스가 명확한가?**
- [ ] 개발 환경 세팅 계획이 있는가? (Docker, CI/CD 등)
- [ ] 배포 전략이 있는가? (Blue-Green, Canary 등)
- [ ] 모니터링/로깅 계획이 있는가?

✅ **기술 부채 관리 계획이 있는가?**
- [ ] "6개월 후 리팩토링이 필요한 부분"이 예측되는가?
- [ ] "기술 부채로 인한 예상 개발 속도 저하"가 정량화되는가?
- [ ] "언제 부채를 갚을 것인가"의 계획이 있는가?

✅ **모든 팀원이 이 기술 선택에 동의할 수 있는가?**
- [ ] 프론트엔드 엔지니어: "이 기술로 개발할 수 있다"
- [ ] 백엔드 엔지니어: "이 스택이 현실적이다"
- [ ] DevOps/인프라 엔지니어: "이 구성을 운영할 수 있다"
- [ ] CTO/기술 리더: "이것이 우리의 기술 방향이다"

✅ **기술 선택이 비즈니스 목표를 지원하는가?**
- [ ] "왜 이 기술을 택했는가?"의 답이 **기술만이 아니라 사업과도** 연결되는가?
- [ ] 예: "마이크로서비스는 장기 운영 비용을 낮춘다"

## 최종 검수 체크리스트

기술 리서치가 완성되었을 때 이것을 확인하세요:

```
조사 단계:
[ ] 각 Branch의 기술 분석이 3개 이상의 사실 기반인가?
[ ] 각 Branch가 다른 기술을 권장하고 있는가? (충돌 확인)
[ ] 구현 가능성이 평가되어 있는가? (우리 팀이 할 수 있나?)
[ ] 성능/보안/확장성이 모두 고려되었는가?

토론 단계:
[ ] 최소 3개 이상의 기술적 트레이드오프가 명시되는가?
[ ] 각 트레이드오프가 정량적으로 비교되는가?
[ ] 실제 사용 사례/실패 사례가 포함되는가?
[ ] 4개 관점이 합의하는 기술이 있는가?

시나리오 단계:
[ ] 3개 시나리오의 기술 스택이 각각 다른가?
[ ] 각 시나리오의 개발 일정이 다른가?
[ ] 각 시나리오의 위험도가 명확한가?
[ ] 실제로 만드는 엔지니어가 보고 "선택할 수 있을" 수준인가?

최종:
[ ] 선택된 기술 스택이 구체적인가? (버전, 구성까지)
[ ] 개발 프로세스가 명확한가? (개발 환경 부터 배포까지)
[ ] 기술 부채 관리 계획이 있는가?
[ ] 팀원 전체가 이 기술 선택을 이해하고 동의하는가?
[ ] "왜 이 기술인가?"를 설명할 수 있는가?
```

## PHASE 1: 기술 심층 조사 (Fork-Based 병렬 탐색)

### 조사 Structure: 각 팀별 2개 Branch 병렬 실행

각 Teammate는 자신의 기술 영역 내에서 **상충하는 2가지 선택**을 동시에 탐색합니다.

#### **Core Tech Researcher - 2개 Branch**

**Branch 1.1: Aggressive Tech Stack (공격적 기술 선택)**

```
당신은 **최신 기술과 가능성을 중시**하는 기술 분석가입니다.

관점: "좋은 기술은 우리를 더 강하게 만든다. 배우는 것은 투자다."

분석 대상: [리서치 주제]

분석 내용:

1. 해당 영역의 최신 기술/프레임워크 분석 (최근 1-2년)
   - 기술 A: [특징, 성능, 커뮤니티]
   - 기술 B: [특징, 성능, 커뮤니티]
   - 기술 C: [특징, 성능, 커뮤니티]
   - 각 기술의 성능 벤치마크 (정량적 수치)

2. 해당 기술을 채택한 **실제 성공 사례** 최소 3개
   - 회사명, 규모, 도메인, 왜 선택했는가?, 결과는?
   - 우리와의 유사도: [높음/중간/낮음]

3. 우리가 이 기술을 써야 하는 이유
   - 성능상 이점: [구체적]
   - 개발 효율성 이점: [구체적]
   - 향후 확장성 이점: [구체적]

4. 우려 사항과 대응
   - 이 기술의 약점: [구체적]
   - 우리가 이것을 극복할 수 있는가?: [YES - 구체적 방법]

결론:
- 강력 추천도: [9/10]
- 우리 팀이 6개월에 습득 가능한가?: [Y/N]
- 채용 시장에서 이 기술자 구하기: [쉬움/어려움/불가능]

| 기술 리더십 필요 | 매우 높음 | 중간 | 낮음 |
| 예상 성공도 | 중간~높음 | 높음 | 매우 높음 |

## 선택 기준:

Cutting Edge를 택할 때:
- [ ] 우리 팀의 기술 역량이 탁월하다
- [ ] 시장이 기술 혁신을 요구한다
- [ ] 장기 운영 비용 절감이 중요하다
- [ ] CTO/리더가 최신 기술에 능숙하다
→ 2개 이상 해당하면 Cutting Edge 가능

Balanced-Tech를 택할 때:
- [ ] 팀이 중간 수준의 기술 역량이 있다
- [ ] 시장이 현실적인 기간을 요구한다
- [ ] 기술과 안정성의 균형이 중요하다
- [ ] 팀이 학습할 의지가 있다
→ 3개 이상 해당하면 Balanced-Tech

Proven Stack를 택할 때:
- [ ] 팀의 기술 역량이 제한적이다
- [ ] 시장이 빠른 출시를 요구한다
- [ ] 안정성과 신뢰성이 최우선이다
- [ ] 장기 유지보수 팀이 필요하다
→ 3개 이상 해당하면 Proven Stack
```

## 팁 & Best Practices

### Branch 전략을 효과적으로 운영하기

**PHASE 1 (조사 Branch) 운영 팁**
- 각 Branch는 **완전히 독립적**으로 (다른 Branch를 보지 말 것)
- 기술 선택의 **근거(벤치마크, 사례, 성능 수치)**를 반드시 포함
- "모든 Branch가 추천하는 기술"을 찾을 것 (진정한 필수 기술)
- **반대 의견이 있으면 좋음** (기술 선택의 논거가 명확해짐)

**Theory Foundation Expert Branch 운영 팁 (신규)**
- 이론 분석은 **단순한 설명이 아니라 학파적 근거**를 제시할 것
- 예: "MVC 패턴"이라고만 하지 말고 "마틴 파울러가 정의한 MVC 패턴의 3가지 원칙"과 그 근거를 제시
- **고전 이론과 최신 이론의 차이**를 명확히 할 것
  - 고전: "왜 이 원칙이 60년 동안 살아있는가?"
  - 최신: "무엇이 바뀌었고, 무엇은 불변인가?"
- **이론과 현실의 갭**을 솔직하게 인정할 것
  - 예: "ACID는 이상이지만, 현실은 결국 타협"
  - 예: "함수형 프로그래밍 이론은 우아하지만, 팀이 이해할 수 있는가?"
- **참고 자료 명시**: 논문/책/저자/발표년도를 구체적으로
- **우리 팀이 이론을 이해할 수 있는가?**를 평가할 것
  - "이 이론의 수학적 기초는?"
  - "팀원이 이해하려면 얼마나 배워야 하나?"

**PHASE 2 (토론 Branch) 운영 팁**
- 각 Branch는 **특정 가치를 극단까지 밀어본다** (최신만, 안정성만, 등)
- 이렇게 하면 "각 가치를 1순위로 할 때 어떤 기술을 선택하는가?"를 명확히 봄
- 4개 Branch 결과를 비교할 때 **기술별 우선순위 변화**를 표로 정리할 것
- "가장 합리적인 기술 조합은?"을 발견할 것
- **이론 기초가 다른 기술들의 선택**을 토론하기
  - 예: "RDBMS (고전 이론) vs NoSQL (최신 이론)"의 토론에서
  - ACID vs BASE의 이론적 근거

**PHASE 3 (시나리오 Branch) 운영 팁**
- Cutting Edge, Balanced, Proven Stack은 **철학적 선택**
- 각 시나리오는 "이 길을 가면 6개월 후 이렇게 된다"를 명확히 보여줄 것
- **개발 난이도, 팀 학습 곡선, 예상 버그**를 정량화할 것
- 선택 기준을 명확히 할 것 (팀 역량, 시장 요구, 리소스 제약 등)
- **각 시나리오의 이론적 근거**가 다양한가?
  - Cutting Edge: 최신 이론 추종
  - Balanced: 고전+최신 이론 균형
  - Proven Stack: 고전 이론 중심

**PHASE 4 (최종 통합) 운영 팁**
- 3개 시나리오 중 "최고"를 고르는 것이 아니라 **"우리 팀 상황에 맞는" 것을 고르기**
- "이 시나리오를 택하지 않은 이유는?"를 기록할 것 (미래의 재검토 기준)
- 최종 로드맵에 "선택 근거"와 "버린 시나리오"를 명시할 것
- **팀 전체가 서명하는 것이 중요** (앞으로 6개월 이것이 팀의 기술 방향)
- **선택된 기술의 이론적 근거를 팀이 이해**하도록 교육 계획 수립

### 기술 리서치 특화 팁

- **성능 수치는 필수**: "좋다"는 추상적 주장 금지, 벤치마크 수치 제시
- **실제 사례 중심**: 이론이 아니라 "이 기술로 만든 회사들의 사례"로 설득
- **우리 상황과의 유사도**: 예를 들어 "Facebook이 쓴다"는 중요하지 않음, "우리와 같은 규모의 한국 회사가 쓰고 있는가?"가 중요
- **학습곡선 현실성**: "배우기 쉽다"는 누가 배우는가? 팀의 기술 수준에 맞는가?
- **채용 현실성**: "이 기술 인력을 구할 수 있는가?" 현지 시장을 고려해야 함

## 예시: 기술 이론 심층조사 (Theory Foundation Expert)

**리서치 대상**: "데이터베이스 기술 선택"
**주제**: "RDBMS vs NoSQL"
**핵심 이론**: "데이터 일관성 이론"

### PHASE 1 - Theory Branch 분석 결과

**Branch 5.1: Modern Theory (최신 이론 기반)**

```
NoSQL의 이론적 기초:

1. CAP 정리 (Eric Brewer, 2000)
   - 저자: Eric Brewer (UC Berkeley)
   - 핵심: 분산 시스템에서는 Consistency, Availability, Partition-tolerance 중 2개만 선택 가능
   - NoSQL 철학: AP를 선택하고 결국적 일관성(Eventual Consistency)를 받아들임

2. BASE 이론 (dan Pritchett, eBay, 2007)
   - Basically Available, Soft state, Eventually consistent
   - ACID를 극복하기 위한 새로운 이론

3. 최신 발전 (2015-2024)
   - CRDT (Conflict-free Replicated Data Types): 이론적 돌파
   - Event Sourcing + CQRS 패턴: CAP 이론의 실무적 극복
   - 비용-이득 분석: "일관성을 포기하면 성능이 N배 증가"

이론이 기술에 미친 영향:
- MongoDB, Cassandra, DynamoDB 등이 BASE를 따름
- "최종 일관성으로 충분하다"는 가정이 코드 설계를 지배
```

**Branch 5.2: Classical Theory (고전 이론 기반)**

```
RDBMS의 이론적 기초:

1. 관계형 모델 (Edgar Codd, 1970)
   - 저자: Edgar Codd (IBM)
   - 논문: "A Relational Model of Data for Large Shared Data Banks"
   - 핵심: 정규화를 통해 데이터 일관성을 보장

2. ACID 이론 (1980-1990)
   - Atomicity: 모두 성공 또는 모두 실패
   - Consistency: 데이터 무결성 보장
   - Isolation: 동시성 제어
   - Durability: 영구성
   - 40년 검증: 모든 금융/중요 시스템의 기반

3. 정규화 이론 (3NF, BCNF, ...)
   - 불변 원칙: 데이터 중복 제거 = 일관성 보장
   - 이 원칙이 50년간 깨진 적 없음

ACID가 기술에 미친 영향:
- PostgreSQL, Oracle, MySQL 등이 ACID를 엄격하게 구현
- "데이터는 항상 일관적이어야 한다"는 가정이 스키마 설계를 지배

고전 이론의 한계:
- 대규모 분산 환경에서 ACID를 모두 지킬 수 없음
- 성능과 일관성의 트레이드오프
```

**이론과 현실의 갭:**

```
이론적 갈등:
- RDBMS 이론: "일관성이 없으면 재앙"
- NoSQL 이론: "완벽한 일관성은 성능을 죽인다"

현실의 타협:
- 대규모 웹 서비스: NoSQL 사용 (AP 선택)
- 금융 시스템: RDBMS 사용 (CA 선택)
- 하이브리드: "중요한 데이터는 RDBMS, 임시 데이터는 NoSQL"

우리 팀의 선택:
- 고객 주문 정보: ACID 필수 (이론적으로 RDBMS)
- 사용자 활동 로그: 최종 일관성 OK (이론적으로 NoSQL)
- 추천 알고리즘 캐시: 정확성 불필요 (이론적으로 NoSQL)
```

### PHASE 2 - 이론적 근거와 함께 기술 선택

```
4개 관점이 모두 고려한 이론:

1. "이 이론의 근거가 과연 타당한가?"
   - ACID가 진짜 필수인가? → 프로젝트에 따라 다름
   - 최종 일관성이 정말 충분한가? → 비용-이득 분석 필요

2. "이론과 현실의 갭은?"
   - 이론: PostgreSQL은 완벽한 ACID를 보장
   - 현실: 분산 트랜잭션은 여전히 어려움
   - 극복: ACID만 믿지 말고 애플리케이션 레벨 처리

3. "우리 팀이 이론을 이해하고 있는가?"
   - ACID를 정확히 이해하고 있는가?
   - NoSQL의 "최종 일관성"을 프로덕션에서 어떻게 구현하나?
   - 이해 부족시: 이론 교육 시간 필요
```

### PHASE 3-4 - 이론 기반 기술 로드맵

```
선택된 기술:
- PostgreSQL + MongoDB Hybrid

이론적 근거:
1. "고객 주문" (ACID 필요)
   - ACID 이론에 근거: 재무 무결성 필수
   - PostgreSQL의 ACID 보장 활용
   
2. "사용자 활동" (최종 일관성 OK)
   - CAP 이론에 근거: 성능(Availability) > 일관성(Consistency)
   - MongoDB의 샤딩 (Partition-tolerance) 활용

3. "팀 학습 계획"
   - ACID 이론: 6시간 교육 (중요)
   - CAP 이론: 4시간 교육 (중요)
   - 실무 적용: 스프린트 1-2에 이론 기반 설계 리뷰

이론과 현실의 차이 인정:
- "PostgreSQL이 분산 트랜잭션을 100% 보장할 수 없음"
- "MongoDB는 완벽한 샤딩이 아님"
- 각각의 한계를 이해하고 애플리케이션에서 보정
```
